# Post-Migration Verification Checklist

Complete this checklist **after** running `python migrate.py --execute`.

---

## Automated Checks (run by the tool)

These are performed automatically. Review the `migration_report_executed_*.md`
file generated by the tool and confirm:

- [ ] `Files defined in map` matches the count in `migration_map.json`
- [ ] `Errors: 0` — no fatal errors reported
- [ ] Warnings reviewed and understood (missing source files are usually OK if
      they were already deleted in a previous run)

---

## Manual Checks

### 1. Boot the application

```
# Open index.html in the browser (or via your local server)
```

- [ ] Boot screen appears (OOBE or Login)
- [ ] No console errors related to module not found
- [ ] No `Failed to resolve module specifier` errors
- [ ] Login flow completes and desktop loads

---

### 2. index.html — Static Asset References

The tool does NOT rewrite HTML files. Manually verify:

- [ ] Open `index.html` in a text editor
- [ ] `<script type="module" src="...">` entry point path is still correct
- [ ] `<link rel="stylesheet" href="...">` CSS paths are still correct

### 3. sw.js — Service Worker Cache Manifest

The tool excludes `sw.js` from import rewriting (it uses path strings in arrays,
not import statements). Manually update all paths in `ASSETS_TO_CACHE` that
reference moved files:

- [ ] `./src/managers/*` → `./src/platform/<domain>/*`
- [ ] `./src/services/*` → `./src/platform/<domain>/*`
- [ ] `./src/system/environments/*` → `./src/platform/environments/*`
- [ ] `./src/system/themes/*` → `./src/platform/theming/themes/*`
- [ ] etc. — cross-reference `migration_map.json` for the full list

### 4. AppRegistry.js — entryPoint Strings

The `entryPoint` field is a runtime string, not a static import. Update any
provider entry points that reference moved search providers:

- [ ] `../services/providers/FileSearchProvider.js` → `../platform/search/providers/FileSearchProvider.js`
- [ ] `../services/providers/PackageSearchProvider.js` → `../platform/search/providers/PackageSearchProvider.js`
- [ ] Verify all other `entryPoint` strings still resolve correctly

### 5. kernel.js — registerSource() String Path

- [ ] Find the `registerSource('builtin-official', ...)` call
- [ ] Update the path string to `../src/platform/packages/repositories/official.json`

### 5. Kernel Import Check

The kernel is the most import-heavy file. Open `src/kernel/kernel.js` and
verify:

- [ ] All imports at the top of the file resolve to `../platform/...` paths
- [ ] No imports still point to `../managers/...` or `../services/...`
- [ ] No imports still point to `../system/capabilities/...`

---

### 6. Application Spot-Check

Open one or two apps in `src/apps/` and verify:

- [ ] Their imports resolve correctly to the new `../platform/...` paths
- [ ] The apps still function (e.g. Settings app opens without errors)

---

### 7. Re-run Automated Verification

```powershell
python tools/migration/migrate.py --verify
```

- [ ] Report shows 0 errors
- [ ] Report shows 0 stale import warnings
- [ ] All destination files confirmed present

---

## If Something Is Wrong

### Stale import found
The import rewriter missed a reference. This can happen with non-standard
import patterns. Fix manually, then run `--verify` again.

### Module not found at runtime
The browser's ES module resolver is stricter than Node's. Ensure:
- Import paths end in `.js` (ES modules require explicit extensions)
- The path capitalisation matches exactly (Windows is case-insensitive, but
  the browser may not be)

### Rollback
See `migration_plan.md` → Rollback Procedure.
